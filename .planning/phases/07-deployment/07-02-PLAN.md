    - Adapters are staggered: first adapter at offset 0, second at intervalMs/N, third at 2*intervalMs/N (evenly distributed within intervalMs)
    - start() sets up timers that call triggerFn(adapterName) at staggered intervals
    - stop() clears all timers
    - All timers use unref() to prevent blocking process exit
    - With 2 adapters and 60min interval: eos at :00, emhass at :30
    - With 3 adapters and 60min interval: eos at :00, emhass at :20, evcc at :40
    - createComposeLifecycle({ composePath, profile, log }) returns { start, stop, getManager }
    - start() creates a compose-manager and calls up(), logs result
    - stop() calls down() on the manager, logs result
    - start/stop swallow errors and log warnings (non-fatal if Docker unavailable)

Create `dvhub/modules/optimizer/services/staggered-scheduler.js`:

Factory function `createStaggeredScheduler({ adapters, triggerFn, intervalMs = 3600000, log })`.
- `adapters` is an array of adapter name strings (e.g., ['eos', 'emhass'])
- Calculate stagger offset: `intervalMs / adapters.length` per adapter
- `start()`: for each adapter, create a setTimeout for initial offset, then setInterval for intervalMs. All timers .unref(). Store timer refs.
- `stop()`: clear all timers.
- triggerFn receives the adapter name so the optimizer module can trigger only that specific adapter.

Create `dvhub/modules/optimizer/services/compose-lifecycle.js`:

Factory function `createComposeLifecycle({ composePath, profile, log })`.
- Imports `createComposeManager` from `../../core/compose-manager.js`.
- `start()`: creates compose-manager, calls `await manager.up()`, logs success. On error, logs warning but does NOT throw (Docker might not be installed in native mode).
- `stop()`: calls `await manager.down()`. On error, logs warning.
- `getManager()`: returns the compose-manager instance (for health checks from API routes).
